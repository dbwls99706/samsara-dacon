# 01 — 1등을 위한 전략 (Decision Tree)

> "29팀 중 1등." 평균 이상이 아니라 **명확한 1위**가 되어야 한다.

## 1. 기준선 분석 — 29팀이 만들 가장 흔한 게임 5종

바이브 코딩 해커톤에서 1인 개발자가 5주 안에 가장 흔히 만들 가능성:

1. **Suika(수박) 머지 클론** — 통계적으로 5팀 이상 나올 가능성. 차별화 어려움.
2. **Vampire Survivors 미니** — Phaser/Pixi로 1주 만에 가능. 흔함.
3. **퍼즐(2048, 테트리스, 사목) 변형** — 클론 티가 너무 남.
4. **하이퍼캐주얼 1버튼 점프** — Flappy Bird 변종. 즉시 식상함.
5. **AI/ChatGPT 기반 텍스트 게임** — 키 없이는 작동 안 함 → 실격 위험.

**우리는 이 5개 카테고리에 들어가면 안 된다.** 들어가더라도 한 줄로 차별점이 설명되어야 한다.

## 2. 평가 점수표를 거꾸로 풀기

배점이 가장 높은 걸 절대 깎지 않는 의사결정 트리.

```
완성도/안정성 (25) ←─ 최우선. 기능 1개 줄여서라도 사수.
참신성 (20)        ←─ "한 줄 트위스트"로 확보.
UI/UX (20)         ←─ 디자인 시스템 1개 정해 일관 적용.
기획↔구현 일관성 (20) ←─ 기획서에 쓴 핵심 루프 그대로 구현. 새 기능 X.
재미/몰입 (15)      ←─ 코어 루프 잘 짜면 따라옴. 마지막에 챙긴다.
```

### 25점 사수 — 완성도 체크리스트 (이 모두 통과해야 1등권)

- [ ] 새로고침 후에도 진행 보존 (localStorage)
- [ ] 모바일 Safari/Chrome, 데스크톱 Chrome/Firefox/Edge 4종 정상 동작
- [ ] 콘솔 에러 0건
- [ ] 60fps 유지 (저사양 기기에서도)
- [ ] 로딩 < 2초 (LCP), 첫 입력 < 3초
- [ ] 회전(가로/세로) 변경에 깨지지 않음
- [ ] 게임 오버 → 재시작 흐름 끊김 없음
- [ ] 모든 텍스트 한국어 일관성 (영어 혼용 시 의도된 부분만)
- [ ] 음소거 기본 OFF + 음량 컨트롤 + 즉시 음소거 토글
- [ ] 키보드 + 마우스 + 터치 3종 입력 모두 사용 가능

### 20점 (참신성) — "트위스트 한 줄" 공식

알려진 장르 X에 알려진 메커니즘 Y를 결합한다. 단, X·Y가 같이 쓰인 사례가 거의 없어야 한다. 예:

- 리듬게임 + 타이핑 = "음악 박자에 맞춰 한국어 받침 입력"
- 클리커 + 로그라이트 = "한 회차마다 빌드를 새로 짜는 클리커"
- 머지 + 디펜스 = "타워 디펜스 수업료를 머지로 충당"
- 카드 빌더 + 도트 픽셀 워크라이커 = "장르 혼합 자체가 컨셉"

**선정 기준**: 기획서 한 줄에 컨셉이 떨어져야 한다. "X + Y인데 ___이 신선하다."

### 20점 (UI/UX) — 디자인 원칙

- **Pico8/Game Boy/네오펑크/모노크롬** 등 좁은 팔레트 1개로 일관 (저사양 1인 개발자가 디자인 일관성 확보하는 가장 빠른 길).
- **첫 화면에서 게임이 이미 움직이고 있어야 함** (메인 메뉴 = 데모 플레이).
- **튜토리얼 = 첫 게임 자체** (시작 → 자동 학습 → 첫 사망 → 본격 플레이).
- **폰트 1개, 색 5개 이내, 사운드 7개 이내**로 통제.

### 20점 (기획↔구현 일관성) — 동결 원칙

- **5/23 (기획서 제출 3일 전)에 핵심 루프 동결.** 그 후로는 기획 변경 X.
- 기획서에 쓴 모든 기능을 실제 구현해야 한다 (없으면 "기획대로 안 만든 팀"이 됨).
- 기획서에 쓰지 않은 기능을 구현해도 점수엔 도움 안 됨 → 일관성 점수만 깎임.

### 15점 (재미/몰입) — 코어 루프 설계

```
[입력 → 즉각 피드백 → 점수/진행 → 다음 도전] × 무한 반복
```

- 한 사이클이 **5~30초**여야 함 (10분 = 20~120 사이클). 너무 길면 중독 안 됨.
- 매 사이클 끝에 **명확한 보상** (점수, 별, 코인, 새 능력 해금).
- **메타 진보** (사이클 ↔ 영구 강화) 1축 이상.

## 3. 1차 통과 전략 (대중 투표 컷오프)

29팀 중 상위 10팀에 들어야 한다. 결국 **다른 제출팀의 눈을 사로잡는** 게임이어야 한다.

### 다른 개발자가 좋아하는 신호

1. **첫 인상 5초 안에 "오 이거 뭐지?"** — 메인 화면이 곧 데모
2. **30초 안에 게임 메커닉 이해** — 튜토리얼 없이도 그냥 시작 가능
3. **3분에 "와 이거 어떻게 만들었지?"** — 기술적 세련됨이 보임
4. **시연영상이 잘 만들어져 있음** — YouTube 30초~1분짜리 트레일러
5. **GitHub README가 깔끔** — 코드도 공개되니까 코드 품질도 본다

> **6/4~6/5에 산출물 업로드를 마치고**, 6/8 마감 전에 깔끔한 README + 트레일러 영상으로 다른 제출팀에게 노출시키는 게 1차 통과의 정공법.

## 4. 2차 (최종) 우승 전략 — 내부 심사위원 정성 평가

심사위원은 **"기획서대로 잘 만들어졌고, 직관적이고, 안정적이고, 짧은 시간에 재미있는 게임"**을 본다. 즉:

- 기획서를 **잘 쓰는 게** 50% (심사위원이 기획서를 읽고 게임을 본다 → 일관성 평가)
- 게임이 **버그 없이 도는 게** 30%
- **차별화된 트위스트**가 20%

**기획서가 한 페이지짜리 약속이고, 게임은 그 약속의 정확한 이행이어야 한다.**

## 5. 시간·리스크 관리

5주 = 35일. 1인 개발자에게는 짧다. 시간을 깎는 의사결정:

| 깎을 것 | 이유 |
|---|---|
| 멀티플레이어 | 백엔드·매칭·동기화 — 5주에 안 됨 |
| 3D | 성능·자산 모두 부담 |
| 오디오 자체 제작 | 라이선스 OK인 음원 사용 |
| 자체 폰트 | Google Fonts / 무료 한국어 폰트 (눈누) |
| 자체 일러스트 | 도트/픽셀 / SVG 기하 도형으로 통일 |
| 회원가입 | 익명 점수만 (Supabase 익명 또는 localStorage) |
| 결제·광고·리워드 | 규정 위반이라 검토 자체 불필요 |

| 살릴 것 | 이유 |
|---|---|
| 글로벌 리더보드 (선택) | 반복 플레이 점수에 직결, 키 없이 가능 |
| 매 게임 후 30초 회고 화면 | "한 판 더" 유도, 시연 영상에서도 멋짐 |
| 컬러풀한 효과 (파티클·셰이크) | 코어 루프 강화, 구현 비용 작음 |
| 한국어 + 영어 토글 | 심사위원·해외 평가자 모두 대응 |

## 6. 단일 결정 우선순위 (의심될 때마다 본다)

1. **마감 사수**: 일정 흔들리면 기능을 깎는다. 일정을 늘리지 않는다.
2. **완성도 우선**: 기능 1개 깎아도 안정성을 챙긴다.
3. **기획서 일관성**: 기획서 외 기능 추가 금지.
4. **모바일 우선**: 데스크톱에서 잘 되어도 모바일이 깨지면 점수 박탈.
5. **시각적 인상**: 첫 5초 캡처가 매력적이어야 한다.

## 7. 절대 하지 않을 일 (Anti-pattern)

- 제출 직전에 새 기능 추가
- 기획 단계에서 코드 작성 시작 (5/26 전엔 프로토타입 외 본 구현 X)
- 기획서를 코딩 끝나고 작성 (PDF는 5/26 마감 → 기획이 먼저)
- "이 정도면 되겠지" 로 모바일 미테스트 제출
- 시연영상 제출 1시간 전 녹화
- 라이선스 확인 없이 자산 사용
- 단일 큰 파일에 코드 몰아넣기 (코드 리뷰 점수 깎임)
- 디버그 로그·console.log·alert 잔존
